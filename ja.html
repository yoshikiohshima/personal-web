Yoshiki Ohshima (Ph.D.)

こんにちは！

コンピュータ・サイエンスの研究とソフトウェア開発をしている大島芳樹です。これまで、プログラミング言語やプログラミング環境、そして対話的なアプリケーションの研究開発、とくに教育向けのシステムに携わってきました。学会活動にも積極的にしています。

東京工業大学の学生だったころは、宣言的記述による最適化コンパイラや入れ子トランザクションを持った言語とったプロジェクトをやっていました。それはそれで良かったのですが、思い返すと、広い世界を知らないままかなり狭い視野にとらわれたままだったように思います。そこで、それが大きく変わることとなりました。

以下で触れるように、あるきっかけからアラン・ケイ博士と彼のグループと仕事をするようになりました。そこでは、「そもそもコンピューティングとはなんなのか」、また禅の公案のような「もし目の前にあるコンピュータのハードウェアを全部取り去ってしまったときに残るコンピューティングとは何か」そして「人類にとってコンピューティングとは何を意味するのか」という問題意識を持ち、それについて話をするだけではなく、実際に形あるものを作っていたのです。

私自身はほどほどのプログラマーですが、プログラミングの技術だけでなくいろいろなことについて学びたいと思っています。これまでに得た知識と目的意識が、私が作るものを特徴付けているように思います。

Squeak Smalltalk (1997-)

1997に開かれたOOPSLAという国際会議に、学生ボランティアとして参加しました。運命のいたずらともいうべき些細なきっかけからSqueakチームのJohn Maloenyに、松岡聡教授から紹介されました。松岡先生はシャープから販売されていたZaurusというPDAを皆に見せびらかせており、そこで私にSqueakをZaurusに移植するようにと言ってきたのです。私はその頃はSmalltalkについてはほとんど何も知らず、ただ、Emacsについてはちょっと知っていたために、対話的に書き換え可能なコンピュータ環境とは良いものだ、という程度の認識を持っている程度でした。そこでSqueakに触れて、まさに驚愕しました。システムの全体を一貫性のある動的システムとして構築したものだったわけですから。その後私がコンピュータの分野でやってきたことは、このシステム全体の設計、動作中のシステムの動的な変更、そして教育分野でのコピューティングという、SmalltalkとSqueakの影響を多大に受けていると言えます。

私がSqueakに行った貢献で多分一番重要なのは、多言語化機能を実装したことかなと思います。もっとも基本的なところの変更であり、20年ほど経った今でもまだ使われています。

SqueakのZaurus PDAへの移植 (1998)

Squeakは移植性を非常に重視して設計されており、非力な携帯型のものも含めた幅広いハードウェアで動作するように作られていました。SqueakのZaurusへの移植を1998年に始めたのですが、運良くシャープが一般ユーザーでもアプリケーションを作れるようにプラットフォームを解放し、また新しくリリースされたハードウェアがちょうどギリギリのメモリを搭載してくれていた、ということもあり、移植作業を完了することができました。他の貢献もあり、アラン・ケイ博士にLos AngelesのWalt Disney Imagineering R&Dに所属していた彼の研究グループと、その年はVancouverで開催されたOOPSLAに招待していただきました。

この時に私は人生初めて飛行機の乗り換えをすることになったのですが、サンフランシスコ空港で見事に乗り継ぎに失敗しました。

NetFrontブラウザのZaurus PDAへの移植 (2000)

ACCESSという会社は、NetFrontという軽量ウェブブラウザで携帯機器向け市場を切り開いているところでした。Squeakと同様にNetFrontはプラットフォーム依存のコードをリンクすることにより移植が簡単にできるように設計されていました。私は恥も臆面もなく、Squeakように書いたコードを再利用しNetFrontの移植を行いました。まあ、自分がオープン・ソースプロジェクト用に書いたコードなのだから良いでしょう。ACCESSの人から見たら、「やけにコード書きが早いやつ」のように見えたのかもしれません。

Parks PDA(2000)

Walt Disney Imagineering R&Dでも、当時市場に出てきていた携帯型の電子機器を使って、テーマパーク来場者に新たなサービスを提供できないのか、ということに関心を寄せていました。また同じ頃、シャープはさらに新しいバージョンのZaurusを販売し始めました。この新しいモデルは、1.1カリフォルニアやフロリダの太陽のもとでもよく見えるカラースクリーン、2. Disneyが求める水準を満たすオーディオやアニメーションをするだけの性能、3. そのようなコンテンツを格納できるカードを挿せるスロット、というDisneyが実験をしようとする時に十分な性能を持ったものでした。Squeakの柔軟性により、Johnと私でDisneyのデザイナー達がコンテンツを作成できるようなフレームワークを手早く作ることができました。実際の来場者を対象としたテストは10月の終わりからオーランドで行ったので、チームみなはDisney提携のホテルに2週間ほど泊まりました。現地での実験がスムースに進むようになると、私は手隙となり、Disney Worldのあちこちのパークに行き、R&Dのバッジに威力を発揮してアトラクションの列を迂回して遊びました。

こういう実験はしばしば地元のニュースとして取り上げることがあり、我々も記事になるかなと期待していたのですが、ブッシュ対ゴアの選挙騒動があったために、我々の実験のような楽しい話に割く紙面はなくなってしまいました。

  • The Parks PDA: A Handheld Device for Theme Park Guests in Squeak

Stitch's Photo Phone (2002)

Stitch's Phone Boothは、テーマパークでの対話的なアトラクションを作る実験でした。これは1家族が入れるくらいの部屋で、壁の一面が平面ディスプレイとなっていました。その画面の上を、キャストがそうさするキャラクター(Stitch)が動き回りつつ、キャストがゲストとお話ができる、というものでした。一通り話しをしたところで、画面上のStitchがカメラを取り出して写真を撮るのでうが、その時にセットに隠されていたカメラからの映像を印刷してそれを家族に渡すようになっていました。私は隠されたカメラのデータを取得するコードを書きました。これは第10回Themed Entertainment Association Awardという賞をもらいました。

こちらの実験はAnaheimで行われたので、私は3週間ほどGlendaleからAnaheimまで毎日(しばしば午前0時過ぎに)車で通いました。

Etoys (2003)

Disneyでは、私は正式にはSqueakチームではありませんでしたく(それでもプロジェクトではSqueakを使い、Squeakへの貢献もしていましたが)。9/11のテロはDisneyの業績にも大きな傷跡を残し、その煽りを受けてSqueakチームはレイオフされることになってしまいました。私はSqueakチームよりは少し長くDisneyにのこりましたが、それでも結局レイオフされることになりました。ビザの関係で日本に帰らなくてはならないところではありましたが、伝手を頼って松尾正信博士に紹介していただき、彼の会社であるTwin Sunで働けるように取り計らってもらいました。ただ、仕事の内容はさらにアラン・ケイとキム・ローズが設立したViewpoints Research Instituteのプロジェクト、中でもEtoysに従事する、ということが多くなりました。Etoysはエンド・ユーザー向けのプログラミング環境であり、Squeakの上に構築されていました。このころからEtoysに深く関与するようになり、数年のうちに徐々にリーダー的役割を果たすようになりました。

Etoys for OLPC (2008-)

OLPC (One Laptop Per Child)とは、世界中の子供達を対象とした安価なラップトップコンピュータを開発し、政府を介して一人一台持てるようにする、ということを目標としたプロジェクトでした。我々のグループも最初期からのソフトウェア設計に関する議論に参加して今いた。Squeakをソフトウェア全体の基盤とし、子供達が「作ることで学ぶ」スタイルで使えるようにする、というアイディアも俎上には上がったりもしたのです。ただ、実際にはその路線とはなりませんでしが、我々のグループもかなりの労力を注いで、EtoysがOLPC上のアプリケーションとして使いやすいように改良をしました。OLPCラップトップに付属していたソフトウェアの中には、「実際に何かを作れる」ものがほとんどなかったために、Etoysは一部の先生や生徒にはとても好評でした。

The STEPS Project (2007-2013)

Viewpoints Researchでは、「パーソナル・コンピュータのソフトウェア全体は2万行のプログラムで必要なものの骨子は全て書くことできるはずである」、というアイディアを実証するためのプロジェクトを、NSFから資金提供により、5年計画で取り組みことになりました。ソフトウェア全体というのは、コンパイラ、ネットワークドライバ、メモリ管理システム、グラフィックスシステム、テキストレイアウト、アプリケーション・フレームワークそして種々のアプリケーションまでも含んだものです。これは比較的な大きなプロジェクトで、メンバーそれぞれがいろいろと興味深いアイディアを出し、それらを実装して行きました。私はアプリケーション・フレームワークをやることになりました。私はFunctional Reactive Progrmmingを元にしつつ、例によって動的なコード変更ができるようなものを設計・実装しました。そのフレームワークでは、アプリケーションと部品の間には区別はなく、どのようなアプリケーションも他のアプリケーションに埋め込めるようになっていました。宣言的な記述、そしてアプリケーションと部品の統合によって、少ない記述量ながらも多機能エディターを作ることできました。

  • The GUI framework for the STEP project

GP (2013-)

John Maloney (彼はScratchのメイン・プログラマでした)とSnap!を開発したJens Mönigと3人で、GPという新たなブロック・ベースのプログラミングシステムを作るプロジェクトを始めました。我々3人を合わせればエンド・ユーザー向けシステムの開発経験は相当なものだと言えなくもないのですが、それでも我々から見ると、いまだに真に汎用のシステム、つまり自分自身をすべて記述できるような強力なものが存在しない、という状況があり、それをなんとかしようということが目標だったわけです。Johnが設計と開発の中心でした。私はモジュールシステムを作ったり、仮想機械の64bit化などを行いました。

Kanto (2017)

I have been interested in computer-based collaboration, especially a place where people can make something together and learn from each other. It occurred to me that with screen sharing and a channel in WebRTC, we can create a collaborative environment even for different block-based editors. I modified the Snap! block editor to allow multiple pointers and hook up the WebRTC based collaboration system to have a functional collaborative blocks-eiting environment. The same collaboration framework can be used to make SqueakJS and the web version of GP collaborative as well.

The name of the system is Kanto. The tradition of this kind of collaboration space was started by a system called Kansas, named after a US state that has a large rectangular flat area. Squeak followed its own implementation called Nebraska. As I was working with Jens, I thought it might be fun to pick a flat area in Germany... but when he told me that it would be “Schleswig-Holstein”, a German name probably was not going to fly, and would probably be unpronounceable for many people. I picked the Kanto region in Japan as the name of the system.

  • Kanto: A Multi-participant Screen-Sharing System for Etoys, Snap!, and GP

Shadama (2017)

I got an urge to make yet another particle simulation system where the program can be modified dynamically while a simulation is running. Web browsers have the bindings to the GPU via WebGL to enable some form of General Purpose GPU programming (GPGPU) and JavaScript allows live programming. The result is Shadama.

One could say that it is embarrassing to work on your Ph.D project again... But sometimes it just happens.

JavaScript version of Croquet (2017-)

When Y Combinator Research could no longer support our research activity, David Smith and some of my colleagues, such as Dan Ingalls, Vanessa Freudenberg and Aran Lunzer, formed a startup. The main goal was to have the real time collaboration technology going on the web. The fundamental algorithm was devised years earlier by the original Croquet group. We reimplemented the algorithm in JavaScript, and created an SDK so that a real time collaborative application can be created easily.

While the SDK provides a good foundation, making a larger application needs a higher-level abstraction, especially when the application requires dynamic DOM manipulation and scripting. I designed and created a virtual DOM based application framework (with help from colleagues, certainly). I also developed a shared space application called Q.

  • Croquet Virtual DOM Framework

Books and Other Documents

Japanese translation of “Powerful Ideas in classrooms” (2005)

I helped publish the Japanese translation of “Powerful Ideas in classrooms”. This is a book on Squeak Etoys and it focuses on the powerful ideas that an interactive computing environment like Etoys can teach. The main theme is ideas such as representing gravity in a discrete differential form is easier for children to understand, and with help from a computer, they can actually manipulate it interactively.

Japanese translation of “Inventive Minds” (2020)

I translated the “Inventive Minds” book. However, I was also involved in the original book project. The book is a compilation of essays by Marvin Minsky, and each essay is accompanied by an article written by Marvin's friends and collaborators. Alan wrote an elaborated essay that he wants to have dynamic interactive programming elements. The paper version of the book cannot contain it (obviously) but with John Maloney, we created the “active essay” version of Alan's article. My long time friend Mr. Kazuhiro Abe connected the O'Reilly Japan to me, and I agreed to take on the translation work.
